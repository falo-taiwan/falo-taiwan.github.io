這個示範在測什麼

這不是大型自動化系統，而是一個最小可行測試：讓 Codex 使用 Chrome 外掛開啟 Google News 台灣首頁，保留當下畫面證據，並產出 10 條新聞標題與網址。

對 AI 教學來說，它展示的是「從聊天到行動」的轉換：AI 不只是回答問題，而是可以被授權去讀取網頁、產出可保存的報告。

一句話版

Codex Automation 負責排程，Chrome 外掛負責進入瀏覽器環境；這次先用手動測試驗證能力，下一步才適合包成定期 automation。

原始測試 Prompt

這是可以直接拿去測試 Codex Chrome 外掛的中文指令。

使用 $chrome:Chrome 打開 https://news.google.com.tw/

請幫我：
1. 截圖當下 Google News 台灣首頁
2. 捕捉最新 10 條新聞的標題與網址
3. 根據當時時間，存成對應檔名的 single-file HTML
4. HTML 需包含截圖與 10 條新聞
5. 只做讀取與整理，不登入、不送出表單、不修改網站設定

實際執行流程

1. 開啟 Chrome 透過 Codex Chrome 外掛打開 Google News 台灣首頁。
2. 擷取畫面 截取當下首頁畫面，作為可視化證據。
3. 抓新聞資料 使用 Google News RSS 取得最新 10 條標題與網址。
4. 產出 HTML 把截圖轉成 base64，與新聞清單一起內嵌。
5. 驗證結果 確認 HTML 可解析、內含截圖、新聞數量為 10。

為什麼採用「Chrome 截圖 + RSS 擷取」

第一次測試時，我嘗試直接從 Google News 頁面的 DOM 逐條讀取連結，但動態頁面節點較多，逐一讀取太慢，導致 Chrome 控制端逾時。

因此改成更穩定的 MVP：Chrome 外掛負責提供當下畫面截圖；新聞資料則從 Google News RSS 抓取。這樣同時保留「瀏覽器現場」與「結構化資料」。

教學重點： AI automation 不一定要硬抓畫面上的所有 DOM。穩定的系統設計通常會把「畫面證據」與「資料來源」分開處理，讓流程更可靠。

如何改成定期 Automation

建立一個 Codex automation。

排程：每天早上 9:00 執行。

任務：
使用 $chrome:Chrome 開啟 https://news.google.com.tw/
截圖首頁，擷取最新 10 條新聞標題與網址。
根據台北時間輸出 single-file HTML。

限制：
只做讀取、截圖、整理與本機檔案輸出。
不要登入、不要送出表單、不要修改設定、不要訂閱、不要發布。
若遇到任何需要確認的網站動作，停止並回報。

安全邊界與驗證結果

只讀任務

本測試沒有登入、送出表單、修改設定、訂閱、付款或發布。

單檔交付

截圖已 base64 內嵌，HTML 可單檔保存、上傳、寄送與放到 GitHub Pages。

驗證完成

HTML parse OK、內含 embedded PNG、新聞項目數量為 10。

治理提醒： 若未來要讓 Codex Chrome 進入已登入網站，建議先限制 allowlist，只允許指定網域；讓 AI 做讀取、整理、草擬，涉及送出、刪除、付款、權限變更或公開發布時必須回到人類確認。
