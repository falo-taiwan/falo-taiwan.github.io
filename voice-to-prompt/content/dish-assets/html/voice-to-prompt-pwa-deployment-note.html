Voice-to-Prompt PWA 部署案例

這份文件用本專案的「即時錄音轉文字 PWA」當案例，說明 PWA 的原理、部署檔案、GitHub Pages 路徑，以及它和 Chrome Extension / Built-in AI 的邊界。

一句話

PWA 是可安裝的網頁 App。它讓網頁可以像本地 App 一樣被加入桌面、獨立視窗開啟、快取靜態檔案；但它不是 Chrome 外掛，也不會自動解鎖 Chrome Built-in AI 的 Prompt API 或 Rewriter API。

案例目標

要證明的事

最小的 Web Speech API 即時轉錄工具，可以從一般網頁升級成 PWA Lite，讓使用者安裝成本地 App 型態。

不承諾的事

PWA 不負責語音模型本地化、不處理上傳音檔，也不保證 Chrome Built-in AI 的 LanguageModel / Rewriter 在任何 origin 都可用。

部署後的使用者流程

1. 使用者打開 https://falo-taiwan.github.io/voice-to-prompt/pwa/
2. 瀏覽器讀取 manifest.webmanifest，知道這是一個可安裝 App
3. Service Worker 註冊 sw.js，快取 PWA 需要的靜態檔案
4. 使用者按「安裝此 PWA」或從瀏覽器選單安裝
5. 安裝後以獨立視窗開啟，使用 Web Speech API 即時轉錄

三個核心檔案

pwa/index.html

真正的 PWA 小工具頁。它包含 Web Speech API 即時轉錄、清除、複製到剪貼簿、安裝提示與 Service Worker 註冊。

manifest.webmanifest

告訴瀏覽器這個 App 的名稱、啟動路徑、作用範圍、顯示模式、主題色與 icon。沒有 manifest，網頁通常不會被視為可安裝 App。

sw.js

Service Worker 是 PWA 的本地代理。它負責快取檔案、接住 fetch request，讓 App 在重新開啟時更像本地工具。

icon.svg

提供 App 安裝時使用的圖示。這個 MVP 用 SVG，方便 GitHub Pages 直接提供靜態檔。

最小原理

<link rel="manifest" href="manifest.webmanifest">

<script>
if ("serviceWorker" in navigator) {
  navigator.serviceWorker.register("./sw.js");
}
</script>

這兩段就是 PWA 的入口。manifest 讓瀏覽器知道「這個網站可以像 App」；Service Worker 則讓網站有快取與離線基礎。

GitHub Pages 部署注意

  • 路徑要用相對路徑：本案例放在 /voice-to-prompt/pwa/，所以 manifest 與 service worker 都用 ./ 開頭。
  • scope 要收斂：scope: "./" 代表這個 PWA 只管理 /pwa/ 子路徑，不接管整個 Voice-to-Prompt 網站。
  • GitHub Pages 是 HTTPS：PWA 與 Service Worker 需要 secure context，GitHub Pages 符合這個條件。
  • 更新會受快取影響：修改 sw.js 或 cache name，才能讓瀏覽器較穩定地更新快取。

和 Chrome Built-in AI 的關係

PWA 只是 Web App 的安裝與快取能力，不是權限升級器。若一般網頁拿不到 LanguageModel 或 Rewriter，改成 PWA 通常也不會自動拿到。這也是為什麼本案例把核心功能放在 Web Speech API，而不是把 Chrome Built-in AI 當必要依賴。

本案例最小檢查清單

  1. 打開 PWA URL：https://falo-taiwan.github.io/voice-to-prompt/pwa/
  2. 確認頁面有「開始轉錄、停止、清除、複製到剪貼簿」。
  3. 確認 DevTools Application 中有 manifest 與 service worker。
  4. 用 Chrome 選單或網址列安裝 App。
  5. 安裝後測試麥克風權限與即時轉錄。
教學定位

這份 PWA Lite 是「Web App → PWA → Extension → Local App」教學路線中的第二站。它證明同一個語音轉文字小功能，可以從普通網頁升級成可安裝的本地感工具，但仍保留清楚的能力邊界。
